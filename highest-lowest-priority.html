  1. 2.1.1 Keyboard - All functionality is operable through a keyboard interface.
    • Reasoning: Keyboard accessibility is fundamental for users with motor impairments, as many cannot use a mouse. Non-compliance would completely block access for these users, making it critical for both usability and legal compliance under WCAG Level A, AA and Section 508.
  2. 1.1.1 Non-text Content - Text alternatives for non-text content.
    • Reasoning: Providing text alternatives for non-text content (like images or charts) is essential for users who rely on screen readers. Without this, visually impaired users would miss key information making this a fundamental accessibility requirement, especially under WCAG Level A and Section 508.
  3. 1.4.3 Contrast (Minimum) â€“ Ensure a contrast ratio of 4.5:1 for text.
    • Reasoning: Insufficient contrast makes text difficult to read for users with low vision. This impacts a wide range of users and is frequently cited in legal cases for non-compliance, making it a high-priority requirement under WCAG Level AA.
  4. 4.1.2 Name, Role, Value - Ensure that programmatic states, properties, and values are defined for UI components.
    • Reasoning: Assistive technologies depend on programmatically defined states and properties to interact with user interface components. If these are not defined, users who rely on assistive technologies will be unable to use the website effectively. This is crucial for screen reader functionality and WCAG Level A compliance.
  5. 2.1.2 No Keyboard Trap - Users can navigate out of any component using a keyboard.
    • Reasoning: A keyboard trap can completely block navigation for users relying on keyboards, rendering parts of the site unusable. Avoiding this is essential for usability and compliance with WCAG Level A.
  6. 1.3.1 Info and Relationships - The structure of content should be programmatically determined.
    • Reasoning: Defining the structure of content programmatically ensures that assistive technologies can correctly interpret the relationships between elements, enhancing comprehension for users with disabilities. This guideline is critical for users of screen readers and is required under WCAG Level A.
  7. 2.4.3 Focus Order - Logical focus order for interactive elements.
    • Reasoning: A logical focus order ensures that users navigating via keyboard can move through interactive elements in an intuitive way. Improper focus order can confuse users, particularly those with motor or visual impairments, making this a necessary requirement for usability and WCAG Level A compliance.
  8. 2.4.7 Focus Visible - Keyboard focus is clearly visible.
    • Reasoning: A visible focus indicator is crucial for users who navigate via keyboard, ensuring they can track which element is currently selected. Lack of focus visibility can make navigation impossible, leading to non-compliance with WCAG Level AA.
  9. 1.4.4 Resize Text - Text can be resized up to 200% without loss of content.
    • Reasoning: Users with low vision often need to resize text to read it comfortably. Ensuring that text can be resized without breaking content or layout ensures accessibility for a wide range of users, making it a key requirement under WCAG Level AA.
  10. 1.4.11 Non-text Contrast - Contrast of 3:1 for non-text UI components.
    • Reasoning: Adequate contrast for non-text elements, such as buttons or form fields, ensures that all users, particularly those with low vision, can perceive and interact with the UI. Non-compliance would reduce usability, making this a WCAG Level AA requirement.
  11. 2.4.4 Link Purpose (In Context) - The purpose of each link is clear.
    • Reasoning: Ensuring that the purpose of each link is clear from its context helps screen readers and users with cognitive disabilities understand where the link leads. This is vital for navigation and usability under WCAG Level AA.
  12. 1.4.1 Use of Color - Color is not the only means of conveying information.
    • Reasoning: Relying on color alone to convey information excludes users who are color blind or have low vision. This is essential for accessibility and compliance with WCAG Level A.
  13. 2.4.6 Headings and Labels - Clear and descriptive headings and labels.
    • Reasoning: Descriptive headings and labels help users, particularly those with cognitive impairments or using assistive technology, understand the structure and content of the page. This improves usability and is required for compliance with WCAG Level AA.
  14. 1.3.2 Meaningful Sequence - Ensures that content is presented in a logical and intuitive order.
    • Reasoning: A logical content sequence is essential for users who rely on screen readers or keyboard navigation. If the sequence is disorganized, it can lead to confusion, making this a key requirement under WCAG Level A.
  15. 2.4.1 Bypass Blocks - Ability to skip repetitive content.
    • Reasoning: Allowing users to bypass repetitive content like navigation menus enhances efficiency, particularly for users navigating with a keyboard or assistive technology. This is a requirement for improving usability and compliance with WCAG Level A.

High

  1. 2.4.11 Focus Appearance (Minimum) - Clear visual focus indicators.
    • Reasoning: A visible focus indicator ensures that users, especially those navigating with a keyboard, can clearly identify the element in focus. The lack of this feature impairs usability for many users, particularly those with visual or motor impairments, making it a critical requirement under WCAG Level AA.
  2. 1.4.5 Images of Text - Use actual text rather than images of text.
    • Reasoning: Using text instead of images of text ensures that content is scalable and accessible to users with low vision or those relying on screen readers. Non-compliance can make content difficult to interact with, reducing usability across devices and screen sizes.
  3. 3.2.1 On Focus - No change of context when UI components gain focus.
    • Reasoning: Ensuring that focus does not trigger a context change prevents confusion and enhances user control, particularly for those using screen readers or keyboard navigation. Compliance with this guideline is crucial for maintaining user orientation.
  4. 3.2.2 On Input - No change of context when UI components gain focus.
    • Reasoning: Input changes should not automatically trigger context changes unless explicitly communicated to the user. This guideline helps maintain user control and prevents disorientation, especially for users with cognitive disabilities or those using assistive technologies.
  5. 1.2.2 Captions (Prerecorded) - Provide captions for prerecorded audio content.
    • Reasoning: Captions must be provided for all prerecorded audio content. Essential for users who are deaf or hard of hearing.
  6. 2.5.8 Target Size (Minimum) - Ensuring that interactive elements (like buttons and links) are large enough to be easily clickable or tappable.
    • Reasoning: The size of the target for pointer inputs must be at least 24 by 24 CSS pixels. Ensures accessibility for users with motor impairments.
  7. 2.2.1 Timing Adjustable - Users can adjust time limits for tasks.
    • Reasoning: Users should have the ability to adjust or turn off time limits for completing tasks. Critical for users with cognitive disabilities.
  8. 3.3.2 Labels or Instructions - Clear labels and instructions for form fields.
    • Reasoning: Clear labels or instructions must be provided for form fields. Enhances usability and comprehension.
  9. 3.3.1 Error Identification - Clear identification of input errors.
    • Reasoning: Users must be informed when an input error occurs. Critical for helping users correct mistakes.
  10. 3.3.4 Error Prevention - Ensuring users can review and correct information before submission.
    • Reasoning: Users should be able to review information before finalizing high-impact submissions. Improves usability.
  11. 3.3.3 Error Suggestion - Suggestions for correcting input errors.
    • Reasoning: When users make an input error, suggestions for correcting the error should be provided. Enhances user experience.
  12. 3.3.7 Accessible Authentication - Provide accessible ways for users to authenticate.
    • Reasoning: Enhances usability but may not be critical.

Medium

  1. 1.2.5 Audio Description (Prerecorded) - Audio descriptions for visual content.
    • Reasoning: Provide verbal descriptions of visual elements in prerecorded content. Important for users with visual impairments.
  2. 1.4.2 Audio Control - Users can control audio playback.
    • Reasoning: Users must be able to control audio playback. Prevents interference for users relying on assistive technologies.
  3. 3.2.3 Consistent Navigation - Consistent navigation mechanisms across pages.
    • Reasoning: Navigation mechanisms should be consistent across pages. Helps users understand and predict navigation.
  4. 1.4.12 Text Spacing - Users can adjust text spacing without loss of content.
    • Reasoning: Users should be able to adjust text spacing without losing content or functionality. Improves readability.
  5. 1.3.5 Identify Input Purpose - Programmatically identify the purpose of each input field.
    • Reasoning: The purpose of each input field must be programmatically determined. Essential for assistive technology users.
  6. 1.4.10 Reflow - Content must reflow without losing functionality.
    • Reasoning: Content must be displayed without losing information or functionality and should not require scrolling in two dimensions. Important for mobile users.
  7. 2.5.3 Label in Name - Visible labels must match programmatically defined names.
    • Reasoning: The visible label of a user interface component must be programmatically associated with its name. Enables voice input functionality.
  8. 1.2.1 Audio-only and Video-only (Prerecorded) - Provide alternative formats for audio-only or video-only content.
    • Reasoning: Provide content in formats that can be easily consumed without visual or auditory elements. Helps users who cannot access audio or video content.
  9. 1.4.10 Reflow - Content must reflow without losing functionality.
    • Reasoning: Content must be displayed without losing information or functionality and should not require scrolling in two dimensions. Important for low-vision users who use browser Zoom up to 400%.

Low

  1. 2.2.2 Pause Stop Hide - Users can pause, stop, or hide moving content.
    • Reasoning: Users must have the option to pause, stop, or hide moving content that starts automatically. Important for users with cognitive or visual impairments.
  2. 1.2.4 Captions (Live) - Real-time captions for live audio content.
    • Reasoning: Captions must be provided for live audio content. Important for real-time accessibility but less common.
  3. 2.5.7 Dragging Movements - Alternative methods for drag-and-drop interactions.
    • Reasoning: Ensure that draggable elements can also be activated using alternative methods. Important for accessibility.
  4. 2.5.1 Pointer Gestures - Ability to cancel or undo actions triggered by pointer gestures.
    • Reasoning: Users should be able to perform actions using single-pointer gestures with alternatives provided for those who cannot use complex gestures.
  5. 2.5.4 Motion Actuation - Alternative methods for motion gestures.
    • Reasoning: Users should have an alternative way to perform actions that rely on motion gestures. Ensures accessibility for those who cannot use such gestures.
  6. 2.5.2 Pointer Cancellation - Ability to cancel or undo actions triggered by pointer gestures.
    • Reasoning: Actions triggered by pointer gestures must have a way to cancel or undo the action. Enhances user control.
  7. 2.4.5 Multiple Ways - More than one way to locate pages.
    • Reasoning: More than one way should be available to locate a Web page. Enhances navigation.

Lowest

  1. 2.3.1 Three Flashes or Below Threshold - No flashing content to avoid seizures.
    • Reasoning: Content must not flash more than three times in any one second period. Important for users with photosensitive epilepsy.
  2. 2.4.13 Content on Hover or Focus - Ensure hover/focus content is easily dismissible or intractable.
    • Reasoning: Ensure that content displayed on hover or focus can be dismissed or interacted with easily. Enhances usability but is less critical.
  3. 2.5.6 Concurrent Input Mechanisms - The user should be able to use a mouse and a keyboard at the same time without losing the ability to interact with the content.
    • Reasoning: Ensure that users can perform actions using different input methods, accommodating a wider range of users.
  4. 3.1.1 Language of Page - The default human language of each Web page can be programmatically determined.
    • Reasoning: Helps assistive technologies convey information accurately.
  5. 3.1.2 Language of Parts - The language used in specific parts of content must be identified programmatically.
    • Reasoning: Important for multilingual content.
  6. 3.2.6 Consistent Help - Provide consistent help across the site to assist users.
    • Reasoning: Enhances understanding of functions.
  7. 3.3.7 Redundant Entry - Aims to minimize the need for users to input the same information multiple times.
    • Reasoning: Improves efficiency.
  8. 3.2.4 Consistent Identification - Components that have the same functionality should be identified consistently.
    • Reasoning: Helps users understand how to interact with components.
  9. 4.1.3 Status Messages - Status messages should be programmatically identifiable.
    • Reasoning: Ensures that users are aware of changes without relying on visual cues.
