Accessibility in HTML with Examples & Best Practices

Purpose

Web accessibility is guided by four key principles: perceivable, operable, understandable, and robust. These principles ensure that all users, regardless of their abilities or disabilities, can access and interact with online content effectively. The purpose of the document "Accessibility in HTML with Examples and Best Practices" is to guide developers and content creators in designing accessible HTML components. It provides practical examples and best practices to ensure digital content is usable by everyone.

Web Complaince

Web accessibility compliance involves adhering to legal standards and guidelines that ensure digital content is accessible to all users, including those with disabilities. Laws like the Section 508 mandate specific accessibility criteria, while the Web Content Accessibility Guidelines (WCAG) outline best practices for creating accessible content. Compliance helps organizations avoid legal issues and fosters inclusivity, enhancing the user experience for a diverse audience.

Note*: The following documentation does not cover all of the WCAG AAA requirements. For all compliance, you can visit the provided links given in front of all four major principles: Perceivable, Operable, Understandable, and Robust.

    • 1.1.1 Non-text Content - Level A All non-text content that is presented to the user has a text alternative that serves the equivalent purpose, Non-text Content Demo Link
      • 1.1.1 - meaningful-image-name All meaningful images that are presented to the user have equivalent, accessible descriptions.
      • 1.1.1 - decorative-image All decorative images that are presented to the user do not have alternate text.
      • 1.1.1 - decorative-background-image requires that important information provided by background images is also included by other methods.
      • 1.1.1 - captcha-alternative requires that CAPTCHA provides formats for users without vision and users without hearing.
  • Guideline 1.2 - Time-based Media requires that alternatives are provided for all time-based media (audio-only, video-only, audio-video, audio and/or video).
    • 1.2.1 Audio-only and Video-only (Prerecorded) requires text or audio alternatives for audio-only and video-only content, ensuring the same information is accessible.
      • 1.2.1-audio-transcript-text requires transcripts are provided for pre-recorded audio-only content. Transcripts must be accurate and complete.
      • 1.2.1-video- alternative-equivalent requires text or audio alternatives for prerecorded video-only content, conveying the same information as the video.
    • 1.2.2 Captions (Prerecorded) requires synchronized captions for all prerecorded audio content, ensuring accessibility for users.
      • 1.2.2-captions-equivalent requires that accurate captions are provided for pre-recorded synchronized media. Captions must be time-synchronized and equivalent.
    • 1.2.3 Audio Description or Media Alternative (Prerecorded) requires that prerecorded media content includes audio descriptions or alternative text tracks.
    • 1.2.4 Captions (Live) requires accurate captions for live synchronized media. The captions must be time-synchronized and equivalent.
  • Guideline 1.3 - Adaptable ensures that the content can be presented in different ways without losing meaning.
    • 1.3.1 Info and Relationships ensures that information and its connections are programmatically identifiable, allowing assistive technologies to convey structure accurately.
      • 1.3.1-heading-determinable and levels ensures that headings are used correctly and in a hierarchical manner, allowing users to easily understand the content.
      • 1.3.1-list-type requires that all visually apparent lists are programmatically identified as the correct list type: ordered, unordered, or description list.
      • 1.3.1-programmatic-label requires that all form elements have programmatic associations which accurately describe the form element and include all relevant instructions.
      • 1.3.1-table-identification requires that data tables have the correct programmatic table markup to identify them as data tables.
      • 1.3.1-cell-header-association requires that data table cells are correctly associated with the relevant headers.
      • 1.3.1-layout-table-structure requires that layout tables are not coded as data tables. No data table semantics should be detected.
    • 1.3.2 Meaningful Sequence requires that content is presented in a logical order, ensuring that users can understand and navigate the information effectively.
      • 1.3.2-content-order-meaning-CSS- position ensures that the visual layout does not change the meaning of content, allowing assistive technologies to interpret it correctly.
    • 1.3.3 Sensory Characteristics requires that instructions for user interactions do not rely solely on sensory characteristics, such as shape, color, or sound.
    • 1.3.4 Orientation mandates that content must be presented in a way that supports both portrait and landscape orientations.
    • 1.3.5 Identify Input Purpose requires that users are informed about the expected purpose of input fields through clear labels or instructions.
  • Guideline 1.4 - Distinguishable ensures users can easily perceive and differentiate content, emphasizing color contrast, text spacing, and visibility to enhance accessibility.
    • 1.4.1 Use of Color requires that color is not the only visual means used to convey information, indicate an action, prompt a response, or distinguish a visual element.
    • 1.4.2 Audio Control requires a method for pausing and stopping audio or controlling the volume of auto-playing content within the first three elements a user encounters on a web page.
    • 1.4.3 Contrast (Minimum) ensures that text and interactive elements have sufficient contrast against their background to enhance readability for users with visual impairments.
    • 1.4.4 Resize Text requires that there is a mechanism to resize, scale, or zoom in on the text to at least 200% of its original size without loss of content or functionality.
    • 1.4.5 Images of Text requires that images of text are either essential, such as logotypes, or presented along with user controls to customize font, size, color and background.
    • 1.4.10 Reflow ensures that content can adapt to various screen sizes, with a minimum width of 320 pixels and a height of 256 pixels.
    • 1.4.11 Non-text Contrast requires that visual elements, such as user interface components and graphical objects, have a contrast ratio of at least 3:1 against adjacent colors
    • 1.4.12 Text Spacing ensures that users can adjust the spacing between characters, words, lines, and paragraphs without loss of content or functionality.
    • 1.4.13 Content on Hover or Focus requires that any content displayed on hover or focus (like tooltips or menus) must be accessible and not obscure other content.

Operable:

  • Guideline 2.1 â€“ Keyboard Accessible mandates that all interface functions are operable via keyboard, ensuring accessibility for users with disabilities.
    • 2.1.1-keyboard-access requires that all functionality of a webpage is accessible through a keyboard, ensuring users can navigate and interact without a mouse.
    • 2.1.2 No Keyboard Trap requires that there are no keyboard traps on the page and users can easily navigate away from any component using only the keyboard.
    • 2.1.4 Character Key Shortcuts requires that character key shortcuts do not interfere with standard keyboard operations, allowing users to navigate without unintended conflicts.
  • Guideline 2.2 - Enough Time ensures that users have sufficient time to read and interact with content.
    • 2.2.1 Timing Adjustable requires that any time limits can be adjusted or extended, allowing users to have adequate time to complete tasks based on their individual needs.
    • 2.2.2 Pause, Stop, Hide requires users to be able to pause, stop, or hide moving content, allowing them to control their viewing experience.
      • 2.2.2-blinking-moving-scrolling requires that users can pause, stop, or hide content that blinks, moves, or scrolls, ensuring they can manage distractions and focus.
      • 2.2.2-auto-updating mandates that users can pause, stop, or hide content that updates automatically, allowing them to focus on the information.
  • Guideline 2.3 - Seizures and Physical Reactions seeks to prevent triggering seizures by avoiding content that flashes more than three times per second and providing warnings.
    • 2.3.1 Three Flashes or Below Threshold requires that content does not flash more than three times in any one second period, reducing the risk of seizures.
  • Guideline 2.4 - Navigable ensures users can easily navigate content through clear options, consistent layouts, and meaningful headings, enhancing the overall browsing experience.
    • 2.4.1 Bypass Blocks requires that users can bypass repetitive content, such as navigation menus, to easily access the main content of a page.
    • 2.4.2 Page Titled mandates that each page has a descriptive title, helping users understand the content and context when navigating through a site.
      • 2.4.2-page-title-defined and purpose requires that each page has a clear and descriptive title that conveys its purpose, aiding users in understanding the content.
    • 2.4.3 Focus Order mandates that the focus sequence of interactive elements follows a logical and intuitive order, ensuring users can easily navigate through content.
    • 2.4.4 Link Purpose (In Context) requires that the purpose of each link is clear from its context, allowing users to understand where the link will take them.
    • 2.4.5 Multiple Ways requires that there are two or more ways (site maps, search, navigation menus, etc) to locate a web page within a set of web pages.
    • 2.4.6 Headings and Labels requires that headings and labels are used effectively to organize content, helping users understand the structure and navigate the information easily.
      • 2.4.6-heading-purpose mandates that headings accurately convey the content they represent, helping users understand the structure and context of the information on a page.
      • 2.4.6-label-descriptive requires that all labels are clear and descriptive, ensuring users can easily understand the purpose of each input element within a form.
    • 2.4.7 Focus Visible mandates that the current focus state of interactive elements is clearly visible, allowing users to easily identify where they are within the interface.
    • 2.4.11 Focus Not Obscured (Minimum) requires that focused elements are not hidden or obscured by other content, ensuring users can easily interact with them.
    • 2.4.13 Focus Appearance (AAA) requires focus indicators with a minimum outline thickness of 2px and a contrast ratio of at least 3:1 against the background for better visibility.
  • Guideline 2.5 - Input Modalities ensures users can interact with content through multiple input methods, such as keyboard, mouse, touch, and voice, enhancing accessibility.
    • 2.5.1 Pointer Gestures mandates that all functions accessible via pointer gestures can also be performed with a single pointer activation, enhancing accessibility.
    • 2.5.2 Pointer Cancellation requires that users can cancel or abort actions initiated by pointer input, preventing accidental activations and enhancing usability.
    • 2.5.3 Label in Name mandates that the name of user interface components includes their visible label, ensuring that assistive technologies can accurately convey their purpose.
    • 2.5.4 Motion Actuation requires that functionality triggered by device motion can also be operated through other means, ensuring accessibility for users.
    • 2.5.7 Dragging Movements mandates that functionality available through dragging gestures can also be accomplished through alternative methods, ensuring accessibility for users.
    • 2.5.8 Target Size (Minimum) mandates that interactive elements have a minimum target size of 24x24 pixels to ensure they are easily clickable or tappable, enhancing accessibility.

Understandable:

  • Guideline 3.1 - Readable focuses on ensuring that content is presented in a way that is easy to read and understand.
    • 3.1.1 Language of Page equires that web pages programmatically identify the pageâ€™s default human language.
    • 3.1.2 Language of Parts requires any part of a page that uses a language other than the pageâ€™s default human language be programmatically identified.
  • Guideline 3.2 - Predictable ensures that user interfaces behave in a consistent and expected manner, allowing users to anticipate how elements will function.
    • 3.2.1 On Focus requires that no unexpected changes of context occur when an element receives focus.
    • 3.2.2 On Input requires that unexpected changes of context do not occur upon changing of form field values or selections.
    • 3.2.3 Consistent Navigation requires that navigation mechanisms remain consistent throughout a website or application.
    • 3.2.4 Consistent Identification mandates that user interface components are consistently labeled and identified across a system, helping users recognize their functions more easily.
    • 3.2.6 Consistent Help requires that help information and support mechanisms are consistently provided across a system, ensuring users can easily find assistance.
  • Guideline 3.3 - Input Assistance ensures users receive support in entering information accurately through clear labels, instructions, and error identification.
    • 3.3.1 Error Identification mandates that errors are clearly identified and described when they occur, ensuring users are aware of issues and can take corrective action.
    • 3.3.2 Labels or Instructions requires that all form elements include clear labels or instructions, helping users understand how to interact with them and improving accessibility.
      • 3.3.2-label-provided mandates that all form elements have clear and descriptive labels, ensuring users can easily understand the purpose of each field for better accessibility.
    • 3.3.3 Error Suggestion requires that when users make errors, the system provides suggestions for corrections, helping them easily rectify mistakes and enhancing overall usability.
    • 3.3.4 Error Prevention (Legal, Financial, Data) requires that users are prompted to confirm actions that could have consequences, such as legal, financial, or data-related transactions.
    • 3.3.6 Error Prevention (All, AAA) mandates that users are provided with options to review and confirm inputs before final submission, helping prevent errors in critical tasks.
    • 3.3.8 Accessible Authentication (Minimum) requires accessible authentication methods, including CAPTCHA alternatives and clear labeling of required information.

Robust:

  • Guideline 4.1 - Compatible ensures that user interface components and content are compatible with assistive technologies, enabling reliable interpretation and interaction.
    • 4.1.2 Name, Role, Value mandates that all user interface components have programmatically determined names, roles, and values.
      • 4.1.2-frame-title requires that all frame elements have a title attribute, ensuring users and assistive technologies can understand the purpose of the content within the frame.
      • 4.1.2-iframe-name mandates that all iframe elements have a title attribute, providing context to users and assistive technologies about the content within the iframe.
      • 4.1.2-change-notify-auto requires that users are automatically informed of changes in content or context, ensuring that assistive technologies can alert them.
    • 4.1.3 Status Messages requires that status changes are programmatically communicated to users, ensuring assistive technologies can announce important updates.
